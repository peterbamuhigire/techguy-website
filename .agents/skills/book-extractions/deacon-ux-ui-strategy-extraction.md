## 1. UX History Timeline (quotable for decks)

Deacon's strongest unique contribution is a multi-millennium history. Useful for client-education slides and strategy-doc context.

| Era | Figure / Concept | Contribution |
|---|---|---|
| 4000 BC | Feng Shui | Arrangement of objects (furniture, layout, color, materials) for energy flow — first user-friendly spatial design |
| 500 BC | Hippocrates / Ancient Greeks | Documented surgeon-workplace ergonomics: lighting, surgeon position (sitting/standing comfortably), tool placement (easy to reach, must not obstruct) |
| Early 1900s | Frederick Winslow Taylor | "Principles of Scientific Management" (1911) — systematic management for human-tool efficiency. Reductive but seminal. |
| 1940s | Toyota Production System | Human-centered: workers can pull a cord to halt assembly line for feedback — usability testing in action. Built on respect for the individual. |
| 1955 | Henry Dreyfuss | "Designing for People" — coined the principle that the designer fails when the product↔person contact becomes a *point of friction* |
| 1966 | Walt Disney | Disney World as immersive UX. Principles: recognize your audiences, fit into their shoes, communicate with color/shape/form/texture |
| 1970s | Xerox PARC | Graphical user interface, mouse — invented personal computing UX |
| 1984 | Apple Macintosh | First mass-market PC with GUI |
| 1995 | Donald Norman (at Apple) | Coined the term "user experience design" — first person with that job title. Wanted broader scope than "human interface" or "usability." |
| 1988 | Norman, *The Psychology of Everyday Things* (later renamed *The Design of Everyday Things*) | Foundational text |

> "User experience includes a various aspect of the end-users relationship with the company, its products, and its services." — Don Norman

## 2. Three Levels of UX Scope (unique framing)

Deacon distinguishes UX scope at three escalating levels. Useful when mapping responsibilities across product + service + brand work.

1. **Single Interaction Level** — experience with a *single product/device for a specific task*. The level where most UX designers work: building an interface for a website or app. Examples: receiving help on phone, filing a claim on insurance website.
2. **Journey Level** — experience across multiple interactive channels/devices to achieve a *goal over time*. Example: receiving an email confirmation, then checking the mail.
3. **Relationship Level** — overall experience with an organization. Aka customer experience level. Examples: researching → buying → using → subscribing.

**Application:** When scoping a UX engagement, declare which level(s) are in scope. Mismatched expectations between client (Relationship) and designer (Single Interaction) cause most "scope creep" problems.

## 3. Five Sub-components of Usability

Standard but worth restating because Deacon explicitly enumerates:

1. **Learnability** — how easy on first encounter; basic tasks must be easy
2. **Efficiency** — speed of task performance after the learning stage
3. **Memorability** — can the user remember features after first use, even infrequent use
4. **Errors** — not error count; error *handling and response*. System should address errors with patience and politeness.
5. **Satisfaction** — system drives the user, not the reverse. Smooth navigation = satisfactory design.

## 4. UI Design Considerations (verbatim list)

When designing any interface, consider:
- **Consistency** — colors, borders, font sizes, styles, backgrounds, effects. Doing similar tasks in similar/familiar ways. Logo always returns to home.
- **Responsiveness** — confirm submissions; loading indicators; never leave the user with a blank page or no path back.
- **Familiar words** — sign-up, register, login. Don't invent new vocabulary that minimizes learning cost.
- **Streaming** — every page contains a link to contact information. Throw away unwanted content; it doesn't make the interface better.

## 5. UI Design Process (7 numbered steps — useful as a starter checklist)

1. Study the idea of the products and design requirements
2. Research potential end-users; study and analyze them
3. Locate a group of people matching end users (5–10 typical, more or less depending on app size)
4. Create use cases and test cases
5. Create paper demonstration after design need is well understood
6. Test and evaluate with possible end-users
7. Create advanced dynamic graphical demonstrations (Photoshop mockups for real-feel)
8. Implement the design (broken into smaller pieces/levels/versions)

## 6. Wireframing Process Variants (useful triage)

Deacon offers three flow patterns to choose from based on project resources:

- **Wireframe → Interactive Prototype → Visual → Design Sketch → Code** (full investment)
- **Sketch → Wireframe → Hi-Def Wireframe → Visual → Code** (medium)
- **Sketch → Wireframe → Visual → Code** (lean)
- **Sketch → Code** (skip visual entirely — appropriate for back-end admin interfaces with low visual weight)

**Heuristic:** "If the resources and time committed are worth the while and business value is high, invest in high-definition wireframe + interactive prototype testing. Otherwise, skip-to-code is acceptable for back-office surfaces."

### 6-step wireframe production
1. **Research** — analyze related products, existing UX trends, best practices, core design guiding principle
2. **Make the research a reference point** — cheat sheet of business + user objectives, personas, use circumstances, competitor cues, audience quotes
3. **Map out user flow** — how many screens, where users come from (which marketing channel, which message), where they end up
4. **Draft, don't draw. Sketch, don't illustrate** — Sharpie on paper. Don't worry about color/aesthetics yet.
5. **Add detail and get testing** — top-to-bottom, left-to-right. Tools: Usability Hub for preference tests.
6. **Turn wireframes into prototypes** — Sketch + InVision; or Figma; or Proto.io / Adobe XD / Framer

## 8. UX Design Principles List (Deacon's specific framing)

Several reinforce common heuristics; the **distinct ones**:

- **Meeting the user's needs** — users-first; what seems brilliant to the org might not be to the user
- **Know your current stage in the designing process** — explicitly tracking process state to avoid getting overwhelmed
- **Well-defined hierarchy** — Two types: (1) content hierarchy, (2) graphical/visual hierarchy. Both required.
- **Simple metaphor** — UX-focused design avoids ambiguous terms; uses simple words
- **Digestibility** — divide large content into small pieces (cognitive-load analogy: chew food)
- **Clarity** (Steve Krug quote: "As a user, I should never have to devote a millisecond of thought to whether things are clickable or not")
- **Familiarity** — complement to clarity. Creativity is welcome, but disregard if it creates confusion.
- **Data-informed design** — qualitative + quantitative from users, marketing, competitors, site analytics
- **Validate designs** — expert review, cognitive walkthrough, formative + summative usability testing
- **Technology flexibility** — orgs must be ready to adjust to trending design and tech

## 9. UX Design Development Steps (6-step lifecycle)

1. **Understanding user needs** — personas; objectives, goals, challenges
2. **Carrying out research** — surveys, interviews; don't assume
3. **Sketching** — outline user journey
4. **Design** — incorporate sketches into a fully-developed website
5. **Implementation** — grant users complete access to the UX design
6. **Evaluation** — pages-viewed and time-on-site as success metrics

## 10. UX vs UI — Body Analogy (memorable for decks)

Deacon's metaphor for explaining the difference:
- **Bones** = code; the structure
- **Organs** = UX design; optimizing/providing input for life processes
- **Appearance, senses, stimuli, reactions** = UI design

**Key claims:**
- UX comes first in product development; UI follows
- UX applies to any product/service; UI is specific to digital
- UI is a **subset** of UX design
