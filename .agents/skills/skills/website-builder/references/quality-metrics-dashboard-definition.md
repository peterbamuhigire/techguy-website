# Quality Metrics Dashboard Definition

This dashboard defines the health metrics for the agency engine itself.
Quality and profit must be reviewed with the same seriousness as craft.

This uses EOS-style scorecard thinking, E-Myth systemization, and launch/retention metrics
already present elsewhere in the repository.

## Dashboard Purpose

- show whether the engine is getting stronger or weaker
- connect repository evolution to commercial and delivery outcomes
- force recurring review of quality, margin, and operational discipline

## Review Cadence

- weekly: operational pulse
- monthly: delivery and client health review
- quarterly: strategic system review

## Weekly Operational Scorecard

Track a small set of leading indicators:

| Metric | Why it matters | Owner | Target |
|---|---|---|---|
| Active projects on track | Detect delivery drift early | Project manager | Defined per portfolio |
| Stage-gate slippage count | Reveals approval and ops weakness | Project manager | Low and decreasing |
| QA blocker count | Shows launch risk and build quality | QA/launch operator | Low and decreasing |
| Launch incidents this week | Reveals operational instability | QA/launch operator | Zero |
| Proposal volume | Tracks pipeline activity | Strategist / sales owner | Defined per quarter |
| Proposal win rate trend | Measures commercial fit and offer quality | Strategist / sales owner | Improving or stable at target |

## Metric Definitions

### Project Margin

Revenue minus direct delivery cost, reviewed per project and portfolio.

### Delivery Cycle Time

From commercial commitment to launch approval, excluding explicitly paused periods if tracked separately.

### QA Defect Escape Rate

Issues found after launch divided by total issues found.

### Proposal Win Rate

Accepted proposals divided by total proposals sent in the review window.

### Client Satisfaction Signal

Can be a structured score, review sentiment, or interview summary, but it must be documented.

## Rule For Metrics

- keep the weekly scorecard short
- use leading indicators weekly and lagging indicators monthly/quarterly
- if a metric does not drive a decision, remove or replace it

## Dashboard Ownership Rule

Every metric must have:

- one owner
- one cadence
- one source of truth
- one target or expected direction

## Review Questions

At each review ask:

- What is worsening?
- What is improving?
- What root cause is behind the change?
- Does the repository need a standards update because of this?
