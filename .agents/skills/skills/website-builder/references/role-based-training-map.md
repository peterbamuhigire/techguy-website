# Role-Based Training Map

Use this map to onboard operators into the agency engine in weeks, not months.
The purpose is to make quality and ownership explicit so delivery is not trapped in the founder.

This map borrows from:

- *Traction* for accountability, role clarity, and operating rhythm
- *The E-Myth Revisited* and *The E-Myth Manager* for systemized work instead of personality-driven work
- *Principles* for explicit standards, diagnosis, and decision quality

## Core Rule

Every role must know:

- which skills it owns
- which artifacts it produces
- which approvals it needs
- which quality standards it must satisfy

## Role Tracks

### 1. Strategist

Primary responsibility:

- turn raw client inputs into business logic, positioning logic, page logic, and trust logic

Skill ownership:

- `website-builder`
- `agency-positioning`
- `design-reference`
- `brand-alignment`
- `they-ask-you-answer`

Primary artifacts:

- qualification notes
- discovery summary
- website strategy brief
- positioning summary
- sitemap and page goals
- offer and CTA hierarchy

Required approvals:

- Gate 1 Qualified Lead
- Gate 3 Strategy Approved

Quality standards:

- recommendations trace back to business logic
- strategy artifacts are specific enough for design and build
- positioning is differentiated and provable

Training path:

1. Read the strategy and intake references in `website-builder/references/`
2. Practice qualification and strategy briefs on past projects
3. Review competitor and trust synthesis work with a senior operator
4. Lead strategy for a supervised project

### 2. Designer

Primary responsibility:

- convert strategy into differentiated visual direction and design constraints

Skill ownership:

- `design-system`
- `design-reference`
- `brand-alignment`

Primary artifacts:

- style brief
- design direction summary
- component and visual system decisions

Required approvals:

- Gate 4 Design System Approved

Quality standards:

- design choices reflect strategy and trust requirements
- premium direction is intentional, not generic
- accessibility and responsive behavior are not afterthoughts

Training path:

1. Read design direction and trust references
2. Analyze three competitor/reference sets and extract usable direction
3. Produce a supervised design direction summary
4. Lead design-system decisions for one project under review

### 3. Front-End Implementer

Primary responsibility:

- convert approved content, design, and SEO direction into reliable pages and components

Skill ownership:

- `page-builder`
- `design-system`
- `photo-manager`
- `i18n`

Primary artifacts:

- reusable components
- production pages
- implementation notes
- known issues list

Required approvals:

- Gate 5 Content and Page Structure Approved
- Gate 6 Build Approved For QA

Quality standards:

- components match the page goal matrix
- accessibility and responsive behavior are preserved
- implementation remains compatible with downstream deploy and SEO work

Training path:

1. Read page-builder and design-system references
2. Build components from existing pattern guidance
3. Pass a supervised implementation review on responsive and accessibility checks
4. Own one build slice end to end

### 4. SEO And Content Operator

Primary responsibility:

- align search intent, messaging, metadata, trust proof, and reporting signals

Skill ownership:

- `seo`
- `content-writing`
- `they-ask-you-answer`
- `blog-writer`
- `seo-audit`

Primary artifacts:

- SEO brief
- topic and intent map
- metadata plan
- content updates
- measurement notes

Required approvals:

- Gate 5 Content and Page Structure Approved
- Gate 8 Post-Launch Measurement Active

Quality standards:

- intent matches page role
- metadata is complete and accurate
- buyer questions are answered directly
- reporting traces to real signals

Training path:

1. Read the SEO, message hierarchy, and buyer-question references
2. Map search intent for an existing project
3. Review one project's metadata and content plan under supervision
4. Own launch measurement setup and first review

### 5. Project Manager

Primary responsibility:

- keep the machine moving through approved artifacts, dates, owners, and decisions

Skill ownership:

- `website-builder`
- `agency-client-retention`
- `update-claude-documentation` when repo maintenance is involved

Primary artifacts:

- kickoff notes
- approvals log
- stage-gate records
- delivery schedule
- post-launch review schedule

Required approvals:

- Gate 2 through Gate 8 coordination

Quality standards:

- source of truth stays in artifacts, not chat
- approvals are recorded the same day
- launch and review dates are explicit

Training path:

1. Read stage-gate and kickoff references
2. Shadow one project through approvals and launch
3. Run one project status rhythm under supervision
4. Own a full delivery timeline for one project

### 6. QA And Launch Operator

Primary responsibility:

- verify release quality, launch safely, and monitor the review window

Skill ownership:

- `deploy`
- `seo`
- `skill-safety-audit` for repo governance changes

Primary artifacts:

- QA matrix
- pre-launch verification
- deployment record
- rollback record
- post-launch review notes

Required approvals:

- Gate 6 Build Approved For QA
- Gate 7 Launch Approved
- Gate 8 Post-Launch Measurement Active

Quality standards:

- core flows are tested, not assumed
- launch has rollback readiness
- monitoring and review windows are active

Training path:

1. Read deploy references and launch standards
2. Run QA and release checks on a staging project
3. Execute one supervised launch
4. Own the full launch and 7-day review cycle

## Training Progression Levels

Use the same levels across roles:

- `Level 1` - can follow the checklist with supervision
- `Level 2` - can own the work with review
- `Level 3` - can review others and improve the system

## Readiness Checklist

An operator is ready to own a role when:

- [ ] they can name the skills and artifacts for the role
- [ ] they know the relevant stage gates
- [ ] they can explain the quality bar without guessing
- [ ] they have completed a supervised project pass

## Cross-Training Rule

Every role should understand the upstream input it consumes and the downstream handoff it creates.
No role should optimize its own output while damaging the full system.
