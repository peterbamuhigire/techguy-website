# 10-Phase Improvement Plan — techguypeter.com

## Overview

This plan takes techguypeter.com from its current score of 6.6/10 (April 2026 audit) to a target of 9.99/10 across all commercial, technical, and content dimensions. The site already has a strong technical foundation, a distinctive visual identity, and a growing bilingual blog. The gap is commercial: the site reads as a capable freelancer rather than an authority practice, the copy is features-first rather than outcomes-first, and there is no visible post-engagement relationship model. These ten phases address those gaps in the correct sequence — foundation before positioning, positioning before copywriting, copywriting before trust architecture — so that each phase's work compounds the one before it.

Every instruction in this plan is grounded in the skills library at `C:\wamp64\www\website-skills\`. No file in this plan was written from first principles; every recommendation traces back to a specific skill reference.

## Critical Path — Phases That Cannot Be Reordered

**Phase 1 must be first.** Without analytics connected, no subsequent optimisation can be measured. Without the 404 page, any broken links from SEO improvements create dead ends. Without the hreflang fix, the FR strategy is misfiring at the foundational level.

**Phase 2 must precede Phase 3.** Copywriting must serve a declared positioning. Writing benefit-led headlines before the category and audience are decided produces generic copy that will need rewriting again. Position first, then write.

**Phase 3 must precede Phase 4.** The trust architecture (Phase 4) must know what claims it is proving. Testimonial placement follows the claims; the claims must be written in Phase 3 before the proof can be interleaved correctly.

**Phase 5 can run in parallel with Phase 4** — technical SEO and schema work is independent of copywriting.

**Phase 6 must precede Phase 7.** TAYA content must exist before AI search optimisation (direct-answer-first formatting, FAQ blocks) can be applied to it.

**Phase 8 can begin once Phase 3 is complete** — the retainer architecture requires benefit-led copy to describe the packages.

**Phases 9 and 10 are sequential** — the offer architecture (Phase 9) must be defined before it can be polished into the elite finish (Phase 10).

## How to Use This Plan

1. **Read the phase README first.** Each phase README is a 3-minute orientation — what the phase does, what the site scores going in, what it scores coming out, and what must be completed before you start.

2. **Execute numbered files in order within each phase.** The numbers reflect dependency. File 01 in any phase creates the foundation that files 02–05 build on.

3. **Run `npm run build` after completing each phase.** Verify the build passes before moving to the next phase. A failed build blocks all subsequent work.

4. **Update the scores in this README after each phase.** Cross out the "After Phase N" row and add the actual date completed. This turns the plan into a record.

5. **Do not skip phases.** Phases 1 and 2 are prerequisites for everything else. The temptation to jump to Phase 6 (content) or Phase 9 (pricing) without fixing the foundation is the most common failure mode.
