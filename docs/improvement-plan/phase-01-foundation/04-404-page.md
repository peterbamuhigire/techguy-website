# 04 — Create a Branded 404 Page

## Skills Applied

- `cro-audit` — `references/heuristic-checklist.md` section 3.9 (No dead ends)

## Current State

`src/pages/404.astro` does not exist. `dist/404.html` does not exist.

Any broken link, mistyped URL, or removed page returns either a server-level error page or an empty response — neither of which is branded, helpful, or convertible. The site has a growing bilingual blog, multiple slug changes over its history, and a language-prefixed URL structure that creates multiple opportunities for 404s (e.g., a visitor typing `/en/services` instead of `/en/services/`, or following an old link to a renamed blog post).

## Target State

- `src/pages/404.astro` exists and renders a fully branded page using `BaseLayout.astro`
- The page includes: the site navigation, a human message explaining the error, links to the homepage and key pages, and a contact CTA
- The page uses the navy-950 dark hero treatment consistent with the rest of the site
- The HTTP response code is 404 (not 200 — Astro handles this correctly by default for `404.astro`)

## Target Score

Technical SEO: **7.0/10**

## Why This Matters

The `seo-audit` checklist item on Crawlability states: "Server returns proper HTTP status codes (200, 301, 404 — no soft 404s)." Without a `404.astro`, the server may return a 200 status code on a "page not found" condition (a soft 404), which confuses Googlebot and wastes crawl budget. The `cro-audit` checklist item 3.9 states: "There is no dead end — every page has a next step for the visitor." A missing 404 page is a dead end for any visitor who arrives at a broken URL — they have no way to recover within the site.

### Step 2: Check the BaseLayout noindex Prop

The `noindex={true}` prop is passed to prevent the 404 page from being indexed by search engines (a correct practice). Verify that `BaseLayout.astro` handles this prop at approximately line 137:

```astro
{noindex && <meta name="robots" content="noindex, nofollow" />}
```

This line already exists. The prop is correctly handled.

### Step 3: Build and Test

```bash
npm run build
```

After building, check that `dist/404.html` exists and contains the branded page content.

To test locally:
```bash
npm run preview
```

Navigate to any non-existent URL such as `http://localhost:4321/en/this-page-does-not-exist`. You should see the branded 404 page, not a server error.

### Step 4: Verify the HTTP Status Code

The Astro framework serves `404.astro` with a 404 HTTP status code by default when a static build is deployed correctly. After deployment to the production server, verify using:

```bash
curl -I https://techguypeter.com/en/this-page-does-not-exist
```

The response should include `HTTP/2 404`. If the server returns `HTTP/2 200`, the Nginx configuration needs updating. Add to the Nginx server block:

```nginx
error_page 404 /404.html;
```

### Step 5: Add the FR Language Version Link

The link `<a href="/fr/">Version française</a>` in the helpful links section allows French-speaking visitors who land on a broken EN URL to navigate to the FR homepage. This is a good i18n practice.
